# Dashboard & Product Direction — Actionable Notes

## Core Product Philosophy

### Tell a story, not just show analytics

The dashboard should guide users through what happened and what they should do next.

Instead of exposing raw analytics, structure the experience like:

1. Current status
2. What changed
3. Why it changed
4. What needs attention
5. Recommended next action

Goal:

> Even a non-technical marketing employee should immediately understand what is happening.

---

## Dashboard Scope

### Dashboard should only contain high-level metrics

Dashboard = overview.

Campaign = detailed analytics.

Posts = minimal information.

Current direction:

* Dashboard → one-glance health
* Campaign → heavy analytics
* Post → lightweight object used to build campaigns

Avoid putting campaign-level analytics onto the dashboard.

---

## Campaign is the Primary Entity

Treat Campaign as the first-class object.

Posts are simply reusable building blocks.

Campaign contains:

* distribution
* analytics
* attribution
* reporting
* optimization

A campaign may contain:

* one post
* multiple posts

Both should work identically.

---

# Dashboard Improvements

## Priorities

Focus dashboard on:

* Items needing attention
* Campaign health
* Current performance
* Recent changes
* Top recommendations

Avoid overwhelming users.

---

## Move most visualizations into Campaign pages

Dashboard should answer:

> "How are things going?"

Campaign should answer:

> "Why?"

This naturally separates complexity.

---

## Spread information across multiple pages

Instead of very dense pages:

* create more focused navigation
* reduce visual overload
* group related metrics

Observation:

Having more pages with less information each was preferred.

---

# MVP Strategy

Current application contains too many features for V1.

Need a subtraction exercise.

Instead of asking:

> What should we add?

Ask:

> What can we remove?

---

Suggested release structure:

## V1

Only essential features.

## V2

Medium complexity additions.

## V3

Everything else.

---

Prioritize based on:

* implementation difficulty
* customer value
* launch importance

---

# Analytics Philosophy

Analytics should always answer a business decision.

Not:

> Here's a chart.

Instead:

> Based on this chart, you should...

Every visualization should help answer:

* Should I continue?
* Should I stop?
* Should I change something?
* What caused this?

# API Strategy

API is valuable.

Do **not** expose it heavily in MVP.

Instead:

* advertise availability
* lock behind higher pricing tier
* unlock later

API should become:

# Design Principles

* Dashboard = overview only.
* Campaign = deep analytics.
* Every chart should support a decision.
* Remove before adding.
* Progressive disclosure over feature dumping.
* Keep marketing workflows non-technical.
* Organize around user workflow instead of feature list.
* Favor simplicity for launch.
* Hide advanced capabilities behind feature flags.
* Prioritize information hierarchy over additional metrics.
